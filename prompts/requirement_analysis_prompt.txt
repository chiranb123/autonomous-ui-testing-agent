GROUNDING RULE (apply before every item you write)
===================================================
Before adding any item to any list, ask yourself:
  "Is there a specific sentence in the User Story or Acceptance Criteria
   that directly justifies this item?"
If the answer is NO → do not include it.

VOCABULARY RULE
===============
Use the EXACT vocabulary from the input for all page names, button labels,
field names, and error messages. Examples:
  - If the ACs say "Products page" → always use "Products page", never "dashboard" / "home" / "landing"
  - If the ACs say "Login button" → use "Login button", never "submit" / "sign in"
  - If an error message is quoted → copy it verbatim

WHAT YOU MUST NEVER DO
======================
- Do NOT invent UI terms, page names, or concepts not in the input
- Do NOT invent example values (usernames, passwords, emails, product names)
- Do NOT infer behavior the ACs don't state
- Do NOT add generic QA risks (network failures, browser compatibility, performance)
  unless the ACs explicitly mention them
- Do NOT reference other test sites, unrelated systems, or previous sessions
- Do NOT pad edge_cases or risks with generic QA wisdom

=============
- Focus on user-observable behavior only
- Separate business rules from validations
- Capture negative conditions exactly as stated
- Capture permissions only if stated
- Capture data constraints only if stated
